# Finch — Engineering Code Review

A prioritized correctness/reliability review of the Finch control plane (Cloudflare Worker
+ Durable Objects), the Go relay agent, and the Next.js dashboard. **Security is reviewed
separately in `security-and-deploy.md`** — this doc does not duplicate that work.

The raw audit produced ~45 findings, but many are repeat confirmations of the same root
defect (the `recordCall` metrics race alone was independently confirmed eight times). The
table below is consolidated to the **distinct** issues; each row notes where it was found.

## 2. Fix before launch

These break the primary, advertised flows or silently corrupt the product's core data. None are
optional for a usable v1.

1. **Tenant identity reconciliation (#4).** The headline bug: the advertised public MCP URL
   (`<sub>.finchmcp.com`) resolves to an empty DO, so **enroll-then-call is broken end-to-end on
   the primary path** — 403/503 every time. Nothing else matters if calls can't route. Fix the
   subdomain→Clerk-id mapping (and the agent's apex `--hub` default + `EnrollResp.url`) first.
2. **EnrollView fabricated ticket (#2) + the broken/missing install pipeline (#29, #30).** The
   onboarding command a user copies cannot join: fake `tk_` ticket, nonexistent `--id` flag,
   missing `/install` 404, and an un-parsed `join` subcommand. Onboarding is dead on arrival.
   Fix as one workstream: render the hub's real `install`/`ticket`, implement `/install`, and fix
   the agent's flag parsing.
3. **Key minting unreachable + revocation broken (#3, #14).** A key-management product where you
   can neither mint a `finch_` key from the UI nor revoke one (and the wired revoke matches by
   label while auth matches by hash) is not shippable. Wire `KeysView`, mint via the real endpoint,
   and revoke by `Key.id`.
4. **`recordCall` metrics race (#1).** Every dashboard metric — calls, p50/p95/err, traffic
   sparkline, recent calls, activity log — is nondeterministically dropped on the success path
   because the fetch handler never captures `ctx` for `waitUntil`. The dashboard reads empty under
   real traffic. The one-line `ctx.waitUntil` fix appears already staged in the working tree —
   **commit it.**
5. **ACL enforcement gap (#5).** The UI promises "every rule is enforced at the door"; the relay
   never reads `s.acl`. Either implement evaluation or remove the claim before launch — shipping a
   false security/access promise is worse than shipping an honest limitation. (Security review owns
   the threat angle; this is the correctness/honesty angle.)

Strongly recommended in the same pass (reliability of the relay itself):
**machine-name encoding (#11)**, **socket reconnect race (#7)**, and **pending-request drain (#9)** —
together they decide whether a real box stays reachable through normal Wi-Fi flaps.

## 4. Overall health

**Solid.** The architecture is sound and the hard parts are right. The Durable-Object split
(TenantDO for control plane + per-machine ApplianceDO for the WS relay) is a clean, correct
topology for a multi-tenant MCP fronting layer, and the storage model (persist raw, recompute
derived fields in `getState`) is the right instinct. Auth primitives are real (HMAC-signed tickets,
hashed `finch_` keys with `last4` stripping on projection), the request *direction* of the relay
forwards headers correctly, and the DO already uses `ctx.waitUntil` correctly in one place
(`appliance-do.ts:85`) — the patterns to copy exist in-repo. The dashboard component structure
(prop-driven sibling views, a single SPA shell, a bridge route layer) is tidy where it isn't
mock-era. The codebase is small enough that all of this is fixable quickly.

**Fragile.** The recurring failure mode is the **same defect repeated**: a correct pattern exists
but wasn't applied consistently. `ctx.waitUntil` is right in the DO but missing on the fetch path
(#1); headers forward one direction but not the other (#10); names decode at the edge in `api.ts`
but stay encoded in `index.ts` (#11); `markMachine` is the intended liveness authority but
`registerMachine`/`approve` write state behind its back (#8, #25). The second pattern is a
**control plane and a UI that are out of sync with each other** — the dashboard fabricates tickets
and keys client-side and discards the hub's real responses (#2, #3), advertises enforcement the
relay doesn't run (#5), and shows phantom users and frozen "now" timestamps (#13, #15). The single
most dangerous fragility is the **identity split (#4)**: two planes key the same Durable Object from
unrelated names, so the entire product appears to work in dev (slug-safe names, apex host coincide)
and silently fails on the real advertised URL. Net read: the engine is well-built but several
critical wires aren't connected end-to-end — this is a codebase that needs its seams stitched and a
**first CI/test harness (#31)** before launch, not a redesign. The total absence of tests is what
let a dozen "the real value is computed but thrown away" bugs accumulate; that's the highest-leverage
process fix.

## 5. Dependencies considered: partysocket

**Evaluated and DEFERRED.** No surface fits. The box-side relay client is Go (can't use the JS
lib). The hub relay is a hibernatable-WebSocket DO where a reconnecting client lib would defeat
hibernation. The only plausible spot — replacing the dashboard's mount/mutation-only fetch (#27) —
is a read-only liveness poll, where a focus-aware `setInterval` (5–10s) on the existing `refetch`
is far simpler than standing up a WS push tier. Revisit only if a browser→hub live-update socket
becomes a real requirement.
